# PLN-003 — Test con prompt override del SupervisorAgent

## Contesto

L'architettura separa la logica di orchestrazione dai prompt (vedi `PLN-001-report.md`).
I prompt del supervisor sono metodi statici (`OrchestratorPrompts.MainContext.stable()`,
`OrchestratorPrompts.Plan.CreatePlan.stable(state)`, ecc.) chiamati a runtime dentro i nodi
del grafo — non al momento dell'import. Questo li rende sovrascrivibili via monkeypatching
senza modificare il codice sorgente.

L'obiettivo di questo piano è definire come creare test che iniettino un prompt alternativo
nel `SupervisorAgent` per:

- Verificare che il grafo risponda diversamente al variare del prompt di sistema
- Testare scenari edge-case (es. supervisor forzato a produrre sempre un piano vuoto, o a
  scegliere solo un agente specifico) senza dipendere dall'LLM reale
- Aprire la strada a test unitari sui singoli nodi del supervisor subgraph

---

## Meccanismo: monkeypatch di `@staticmethod`

`OrchestratorPrompts.MainContext.stable()` è un `@staticmethod`. In Python si può
sovrascrivere con `unittest.mock.patch.object` oppure assegnando direttamente l'attributo
sulla classe.

Il patch deve avvenire **prima** che `GraphInterface` venga istanziato, ma poiché i metodi
sono chiamati a runtime (non cached), è sufficiente che il patch sia attivo al momento
in cui il nodo supervisor viene eseguito.

## File da creare

| File | Tipo |

| `tests/T006_prompt_override.py` | Nuovo script di test |

le conversazioni inline nello script (stessa modalità di `run_tests()` ma con patch attivo).

## Dettaglio implementazione

### `tests/T006_prompt_override.py`

```python
"""
T006 — Supervisor prompt override
----------------------------------
Verifica che il grafo risponda in modo diverso quando il prompt di sistema
del SupervisorAgent viene sostituito con una variante minimale.

Esecuzione:
    python -m tests.T006_prompt_override
"""

import sys
from unittest.mock import patch
from pathlib import Path

from tests._utils import run_tests, silence

with silence():
    from saferplaces_multiagent.ma.prompts.supervisor_agent_prompts import OrchestratorPrompts
    from saferplaces_multiagent.ma.prompts import Prompt


# ---------------------------------------------------------------------------
# Prompt alternativo — supervisor forzato a produrre sempre un piano vuoto
# ---------------------------------------------------------------------------
def _always_empty_plan_prompt() -> Prompt:
    return Prompt({
        "title": "OrchestrationContext",
        "description": "test override — always return empty plan",
        "command": "",
        "message": (
            "You are a test orchestration agent.\n"
            "Always return an empty plan, regardless of the user request.\n"
            "Do NOT assign any steps to any agent.\n"
            "Respond only with an empty plan."
        ),
    })


# ---------------------------------------------------------------------------
# Messaggi di test — stesse richieste di T003/T004 (richiederebbero agenti)
# ---------------------------------------------------------------------------
MESSAGES = [
    "Retrieve the latest DPC rainfall radar for northern Italy",
    "Run a SaferRain flood simulation for Rome with 50mm/h rainfall",
]


# ---------------------------------------------------------------------------
# Esecuzione con patch attivo
# ---------------------------------------------------------------------------
if __name__ == "__main__":
    result_file = Path(__file__).parent / "result" / "T006.md"

    with patch.object(OrchestratorPrompts.MainContext, "stable", _always_empty_plan_prompt):
        run_tests(MESSAGES, result_file=result_file)
```

---

## Comportamento atteso

Con il prompt `_always_empty_plan_prompt` attivo:

- Il supervisor dovrebbe produrre un piano vuoto per entrambi i messaggi
- Il grafo non dovrebbe invocare `retriever_agent` né `models_agent`
- L'output in `T006.md` mostrerà la catena `request_parser → supervisor_agent →
  supervisor_planner_confirm → final_responder` senza nodi specializzati

Questo permette di verificare che il prompt abbia effettivamente influenzato il routing,
confrontando `T006.md` con i risultati di `T003.md` / `T004.md`.

## Come estendere

Lo stesso meccanismo si applica a qualsiasi altro metodo statico dell'`OrchestratorPrompts`:

```python
# Override del prompt di creazione piano
with patch.object(OrchestratorPrompts.Plan.CreatePlan, "stable", my_custom_stable):
    run_tests(MESSAGES, result_file=result_file)
```

Per i metodi dinamici che accettano `state` come parametro, la firma del mock deve rispettarla:

```python
def _custom_create_plan(state) -> Prompt:
    return Prompt({ ... })

with patch.object(OrchestratorPrompts.Plan.CreatePlan, "stable", _custom_create_plan):
    run_tests(MESSAGES, result_file=result_file)
```

## Note architetturali

- `patch.object` è contestuale: il prompt originale viene ripristinato automaticamente
  all'uscita dal `with` block — nessun effetto collaterale sugli altri test.
- Il meccanismo funziona perché i metodi `stable()` sono chiamati a runtime dentro i nodi
  del grafo, non al momento dell'import o della costruzione del grafo.
- Se in futuro i prompt venissero cachati alla costruzione del grafo, questo meccanismo
  smetterebbe di funzionare: occorrerebbe passare i prompt come parametri a `GraphInterface`.
