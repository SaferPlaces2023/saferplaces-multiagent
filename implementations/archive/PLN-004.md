## §1 — Contesto architetturale

L'architettura F009 definisce che ogni agente deve avere un modulo prompt dedicato sotto `ma/prompts/`, contenente una classe con gerarchia nidificata di metodi statici che espongono versioni etichettate dei prompt.

Attualmente:
- `request_parser.py` contiene una classe locale `Prompts` con il singolo attributo `SYSTEM_REQUEST_PROMPT` (costante stringa)
- `final_responder.py` contiene una classe locale `Prompts` con tre attributi: `FINAL_RESPONSE_PROMPT` (costante), `STRUCTURED_FINAL_CONTEXT` e `FORMAT_FINAL_CONTEXT` (lambda che leggono `state`)

Con F009:
- I prompt diventano metodi `@staticmethod` che ritornano istanze della dataclass `Prompt`
- **Prompt statici** (testo fisso): firma `stable() -> Prompt` — nessun parametro
- **Prompt dinamici** (contenuto compilato a runtime): firma `stable(state: MABaseGraphState, **kwargs) -> Prompt` — ricevono lo state per leggere variabili e compilare il testo
- Ogni metodo `stable()` è la versione **in produzione**; versioni alternative `v001()`, `v002()` ecc. sono per test (versionamento indipendente dalla dinamicità)

## §2 — Implementazione: `request_parser_prompts.py`

### Struttura della classe `RequestParserPrompts`

```
RequestParserPrompts
├── MainContext
│   ├── stable() → Prompt
│   └── v001() → Prompt (versione alternativa per test)
```

### Il prompt `MainContext.stable()`

Incapsula `SYSTEM_REQUEST_PROMPT` da `request_parser.py`. Il prompt instruisce l'LLM su:
- Estrazione dell'intent ad alto livello
- Estrazione delle entità nominate
- Copia del testo originale
- Non inventare informazioni

Il prompt **è statico** (ha sempre lo stesso testo): la firma è `stable() -> Prompt` (senza `state`), perché il contenuto non dipende da variabili di runtime.

### Versionamento

Oltre a `stable()`, è utile una versione `v001()` per test-driven development, es. un prompt minimale per debugging. Questa è una versione alternativa ortogonale al concetto di staticità/dinamicità.

## §3 — Implementazione: `final_responder_prompts.py`

### Struttura della classe `FinalResponderPrompts`

```
FinalResponderPrompts
├── Response
│   ├── stable() → Prompt
│   └── v001() → Prompt
├── Context
│   ├── Structured
│   │   ├── stable(state) → Prompt
│   │   └── v001(state) → Prompt
│   └── Formatted
│       ├── stable(state) → Prompt
│       └── v001(state) → Prompt
```

### I tre prompt originali

1. **`FINAL_RESPONSE_PROMPT`** → `Response.stable()`
   - Prompt di sistema che instruisce l'LLM a generare una risposta coerente
   - Firma: `stable() -> Prompt` (non context-aware)

2. **`STRUCTURED_FINAL_CONTEXT`** → `Context.Structured.stable(state)`
   - Serializza lo stato come JSON, interpolando valori da `state.get('parsed_request')`, `state.get('plan')`, ecc.
   - **Dinamico** (compilato a runtime): riceve `state` come parametro perché il contenuto dipende dai valori attuali
   - Firma: `stable(state: MABaseGraphState, **kwargs) -> Prompt`

3. **`FORMAT_FINAL_CONTEXT`** → `Context.Formatted.stable(state)`
   - Formatta lo stato come testo leggibile, interpolando valori da `state`
   - **Dinamico** (compilato a runtime): riceve `state` come parametro perché il contenuto dipende dai valori attuali
   - Firma: `stable(state: MABaseGraphState, **kwargs) -> Prompt`

### Versionamento

Anche qui, per ogni sezione è utile almeno una versione alternativa `v001()` per test. Ad esempio: `Context.Formatted.v001(state)` ritorna un prompt minimale senza tutti i campi, utile per debugging.

```python
prompt = final_responder_prompts.FinalResponderPrompts.Response.stable()
message = prompt.to(SystemMessage)
```

**Prompt dinamico** (`Context.Formatted.stable(state)`) — riceve `state` a runtime per compilare il contenuto:

## Pattern Implementato: F009

Questa implementazione segue il **F009 (Prompt Organization Architecture)** pattern:

### Caratteristiche del Pattern
- **Layered prompt hierarchy**: Class structure mirrors semantic organization (Agent → Category → Method)
- **Static + dynamic prompts**: Methods distinguish between fixed text (`stable()`) and state-compiled text (`stable(state)`)
- **Versioning**: Each method has `stable()` (production) and `v001()/v002()` (testing) variants
- **Dataclass abstraction**: All prompts wrapped in `Prompt` dataclass with consistent interface
- **Import centralization**: Agents import from dedicated prompt modules, not internal classes

### Benefici Realizzati
- ✅ Centralized prompt management
- ✅ Easy versioning and A/B testing
- ✅ Type-safe prompt handling
- ✅ Simplified agent code (cleaner logic)
- ✅ Testability (mockable/patchable prompts)

## Note Importanti

### Import dal modulo `ma/prompts`

`ma/prompts/__init__.py` ora esporta:
```python
from dataclasses import Prompt
from . import supervisor_agent_prompts
from . import request_parser_prompts        # ← AGGIUNTO
from . import final_responder_prompts       # ← AGGIUNTO
```

### Pattern di chiamata a runtime

Ogni nodo che usa un prompt chiama il metodo `stable()` (o una versione alternativa) a **runtime**: non durante l'import del modulo.

Questo consente:
1. **Patch dinamico nei test** via `unittest.mock.patch.object()` (pattern F008)
2. **Lazy evaluation** del contenuto — per prompt dynamici, il testo viene compilato solo al momento della chiamata, quando `state` è disponibile con i valori attuali
3. **Iterazione veloce** durante il debugging

**Nota importante**: il parametro `state` nei metodi dinamici NON è una "versione alternativa". È un parametro **obbligatorio** che serve a leggere i valori attuali del grafo e compilare dinamicamente il testo del prompt. È orthogonale al concetto di versionamento.
