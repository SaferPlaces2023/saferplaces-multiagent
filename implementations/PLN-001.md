# PLN-001 — Allineamento branch `test-pj002` con `origin/test-pj002` e `origin/main`

## Contesto

Il branch locale `test-pj002` è indietro di 1 commit rispetto a `origin/test-pj002`
(commit `e29e70d` — "prompt module") e `origin/main` contiene 5 commit aggiuntivi
(`03cf7eb`, `0af95f2`, `20f8d20`, `f0e5aed`, `a8be5dd`) non ancora presenti nel branch.

L'obiettivo è continuare il lavoro su `test-pj002` incorporando entrambi gli aggiornamenti.

---

## Comandi da eseguire

### 1. Assicurarsi di essere sul branch corretto

```bash
git checkout test-pj002
```

Verifica di trovarsi su `test-pj002` prima di procedere.

---

### 2. Aggiornare il branch locale da `origin/test-pj002`

```bash
git pull
```

Scarica e integra il commit `e29e70d` ("prompt module") presente su `origin/test-pj002`
ma mancante in locale. Essendo un fast-forward, non genera conflitti.

---

### 3. Portare i commit di `main` su `test-pj002` (rebase)

```bash
git rebase origin/main
```

Riapplica i commit di `test-pj002` sopra `origin/main`, incorporando
`03cf7eb`/`0af95f2`/`20f8d20` (cesium), `f0e5aed` (node lifecycle) e `a8be5dd` (silence useless logs).
Il risultato è una storia lineare e pulita.

> **In caso di conflitti**, git si ferma e segnala i file in conflitto. Procedura:
>
> **a)** Vedere quali file sono in conflitto:
> ```bash
> git status
> ```
> I file in conflitto sono marcati come `both modified`.
>
> **b)** Aprire ogni file e risolvere manualmente i blocchi di conflitto:
> ```
> <<<<<<< HEAD          ← versione da origin/main
> ...
> =======
> ...
> >>>>>>> <commit>      ← versione dal tuo commit
> ```
> Scegli quale versione tenere (o combina le due), poi **rimuovi** i marcatori `<<<<<<<`, `=======`, `>>>>>>>`.
>
> **c)** Dopo aver risolto, marcare il file come risolto e continuare:
> ```bash
> git add <file-risolto>
> git rebase --continue
> ```
> Git richiederà di confermare il messaggio di commit — lasciarlo invariato e salvare.
>
> **d)** Ripetere i passi b–c per ogni commit che genera conflitti.
>
> Per annullare il rebase e tornare allo stato iniziale in qualsiasi momento:
> ```bash
> git rebase --abort
> ```

> **Conflitto rilevato durante questa esecuzione** — commit `5cc6429` su `.gitignore`:
>
> `origin/main` aveva aggiunto la sezione `# logs` con `**/**__state_log__*.json`,
> mentre il commit `5cc6429` aggiungeva `saferplaces-multiagent.code-workspace`.
> Entrambe le righe andavano mantenute. Risolto tenendo tutte e due le aggiunte:
> ```
> docs_pj
> saferplaces-multiagent.code-workspace
>
> # logs
> **/**__state_log__*.json
> ```
> Poi: `git add .gitignore` → `git rebase --continue`

---

### 4. Push del branch aggiornato

```bash
git push --force-with-lease
```

Necessario perché il rebase riscrive la history. `--force-with-lease` è più sicuro
di `--force`: fallisce se qualcun altro ha fatto push sul branch nel frattempo.

---

### 5. Committare la cartella `implementations/`

La cartella `implementations/` è untracked e non verrà inclusa automaticamente nel rebase.
Dopo il push, aggiungila al branch:

```bash
git add implementations/
git commit -m "chore: add implementations plans"
git push
```

---

## Verifica finale

```bash
git log --oneline -10
```

La history dovrebbe mostrare i commit di `main` come base, seguiti dai commit di `test-pj002`.
